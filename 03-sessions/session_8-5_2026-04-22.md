# Session 8.5 — Commentary Render Trim Hot-Fix

**Date:** 2026-04-22
**Wave:** A (urgent — unblocks unhiding Calvin / JFB / Wesley for public render)
**Batch spec:** single-concern frontend render fix. No DB schema changes. No ingestion work.

## Summary

Doctrine A §4.4 requires every commentary voice on a verse page — featured *and* non-featured — to render a ≤50-word trim with a "Read full passage" disclosure. In production the featured voice honored that rule but every card inside the "Show other voices (N)" expansion was dumping full `commentary_text` (often thousands of words per card), which violated the curated-not-exhaustive editorial rule and was the reason 3,164 rows of Calvin / JFB / Wesley commentary had to stay `status='hidden'`. The fix extracted the trim + preview decision into a shared pure helper (`previewForVoice` in `src/components/commentary/truncate.ts`), flipped `FeaturedExcerpt` and `CommentaryCard` to `"use client"` with one matching inline-expand mechanic, and migrated `VotdCommentaryHighlights` and `CurationPanel` off the client `FeaturedExcerpt` import path onto the pure helper module. A Node-test-runner regression guard (`__tests__/truncate.test.ts`, 11 cases, no new deps, uses Node 22's `--experimental-strip-types`) locks the shared contract. Vision v2 §4.4 row 3 was amended to make the "same treatment" language explicit so the render layer can't drift off the doctrine again.

## Why a shared helper rather than duplicated trim

Both card components now call `previewForVoice({ commentaryText, featuredExcerpt, cap })`. Two reasons this matters over inline duplication: (1) it's a testable unit without a React renderer, which lets the regression guard live in `node:test` with zero new dependencies; (2) a future curation-editor change that alters featured-excerpt override semantics touches exactly one function rather than two card components plus a server pre-compute. The featured-voice card passes the real `featured_excerpt`; the "other voices" card passes `null` (curator override never applies to non-featured rows) — the contract is explicit at each call site.

## Why the anchor-link on the featured card was also replaced

Pre-existing wart: the featured card's "Read full passage ↓" was a `<Link href="#voice-{featured.id}" />` anchor — but `CommentarySection` excludes `featured.id` from the `others` array, so the anchor landed on nothing. The batch spec required matching mechanics between featured and non-featured, and since "full text lives only in the collapsed card itself" is the new reality everywhere, inline expand/collapse via local client state is the one mechanic both cards share. Side-benefit: deep links to `#voice-{id}` into the "other voices" disclosure still work because `CommentaryCard` still anchors its `<article id="voice-{id}" />`.
